# Minu‑Lovable-ийг Lovable‑ийн хэрэглээний загварт ойртуулах судалгаа

> Энэхүү баримт нь **олон эх сурвалжийн дүгнэлт**: Lovable‑тай зах зээлийн шинж чанараар адилхан ажиллах туршлагын талаарх баримжаа, платформын өөрийн `PLATFORM.md` болон `.claude/prds/web-builder-phase-1.md`-ийн албан замын зорилгуудын хоорондын зөрүүг судалж бичигдлээ.  
> **Кодонд өөрчлөлт оруулаагүй** — зөвхөн сайжруулалтын урсгал, зорилт, суурь ойлголтууд.

---

## 1. Lovable‑тай юу адилхан байхаар хүлээгдэж байгаа вэ?

Lovable-тай төстэй бүтээх туршлагад ихэвчлэн дараах **хэрэглэгчийн туршлагын элементүүд** багтаж санагдана:

| Хэмжээ | Lovable-д ихэвчлэн очих хүлээлт | Анхаарах гол зүйл |
|--------|----------------------------------|---------------------|
| **Хурдан давталт** | Нэг чат захиалгаар preview шууд орж, дараагийн засвар нь секундын дотор туслагдана | Streaming + sandbox I/O саатуулах үе шатыг багасгах |
| **Найдвартай код** | Бүтэн файл, үл тасрах импорт, build алдаагаар бага “цохилт” | Детерминистик гаралт (module assembler)-т шилжих нь платформын PRD-д нийцэн |
| **Чат‑дээр суурьтай засвар** | “Энийг солих”, “Энд нэмэх” гэх мэт нарийн өөрчлөлт засагдана | Edit mode-д intent → файл хайлт → контекст оруулах урсгал (одоогоор байгаа загвар) илүү нарийван |
| **Амьд урьдчилгаа** | Iframe/Dev server синк, алдааны фидбек | Preview console‑оос алдааг parent руу дамжуулах (repo-д эхэлсэн загвар байна), Vite сааталыг засах |
| **Төслийн үргэлжлэл** | Session дахин орж засварлана | Conversation state / projects API (байгаа)‑ийг бүтэн product flow-той хослуулах |

**Дүгнэлт:** Зөвхөн “нууц үлдээлтэй LLM текст гаргадаг систем”‑ээр Lovable-тай UX тэнцүүлэх нь маш хэцүү. Тэр зөрүүг **детерминистик угсралт**, **хаалттай архитектур**, **баталгаажуулалт**-аар нөхөх нь тогтвортой зам.

---

## 2. Minu‑Lovable‑ийн одоогийн гол онцлог (богино)

Репоны одоогийн онцлогоос гаргалгаа:

1. **`generate-ai-code-stream`** голдуу **чанаргүй текст (XML tag гэх мэт) гарган stream** хийнэ — **LLM‑ийн function/tool calling ашиглахгүй** (ээд мөртөөр тэмдэглэгдсэн).
2. **Edit горимд** **`analyze-edit-intent`** нь structured object (`generateObject`)‑оор хайлтын төлөвлөгөө гаргаж, **`executeSearchPlan`** зэрэг нь файл дотор **детерминистик хайлт** хийнэ — энэ нь Lovable-тай төстэй **“agentic scaffolding”**.
3. **Apply** талд **`apply-ai-code-stream`** пакеж суулгах, файл бичих, Vite‑тэй холбоотой саатлууд боловсруулна.
4. **`PLATFORM.md`** болон **PRD Phase 1** нь урт хугацааны зураглал: **чанаргүй “юу ч гарга” биш**, **`modules/store`, `modules/warehouse` урьдчилан бэлдэх + module assembler**.

**Дүгнэлт:** Lovable-тай төстэй *мэдрэмж* үүгээр өгөхөд давтагдах нарийвчилсан чат засвар сайн; харин *бизнес web builder платформ* болгон тогтоох зорилготой бол PRD-д заасан шилжилт гол ач холбогдолтой.

---

## 3. Ялгаа — зүйлээс үлдээх / нэмэгдүүлэх

### 3.1 Lovable-д ойр “огтлохуйц” давуу талнууд

- **Edit-д зориулсан хоёр давхаргатай загвар**: (1) санаачлах төлөвлөгөө, (2) кодын хүсэлт — энэ нь контекстыг томорч, чанартай байдлыг хангана.
- **Streaming + apply нэг урсгал** — хэрэглэгч “юу болж байна” гэдгийг урсгал байдлаар харна.

### 3.2 Сул тал болон эрсдэл (байдлаас гаргалгаа)

- **Текстээс парслагддаг файл** — модел тасрах, хаалт мартсан файл зэрэг Lovable-с гаргаж болох түр зуурын асуудалтай төстэй. PRD-д `enableTruncationRecovery` зэрэг тохируулгыг үл идэвхжүүлсэн байх тохиолдол бий.
- **`assemble-modules` асемблер** PRD-д “файлуудыг хуулна” гэж тодорхойлсон боловч интерфейсын түвшинг шалгах шаардлагатай — **бодит assembler** бол хараахан бүх шатандаа сууж дуусаагүй байж болох зөрүү.
- **`generate-ai-code-stream` нь асемблер болох** алхам нь Lovable-с **илүү платформууд** болно (илүү deterministic, хангалт үзүүлнэ).

---

## 4. Сайжруулах арга зам (үүлдрэлтэй бус — эхэлж хийх дарааллаар санал)

Доорх нь **кодонд гаргалгүй** зөвхөн урсгал/зохион байгуулалтын санал юм.

### A. Бүтээгдэхүүний аль нэг гол зорилгоо зөв тодорхойлох

- **Хувилбар A — “Lovable clone”**: Ерөнхий React/Vite санал болгож чатлана (одоогийн open‑lovable маягийн хүчтэй текст‑генераци).
- **Хувилбар B — “Монгол бизнес платформ”** (`PLATFORM.md` + PRD): **Модуль + assembler + shadcn constraint** — Lovable-с **гарцын загвар**, **хангалтын горим өөр**.

Төслийн бүх сайжруулалт энэ сонголт дээр **нэг дор биш** төвлөрвөл хурд, чанарт ашигтай.

### B. PRD Phase 1-ийг гол зам болгох (архитектурын нийцтэй сайжруулалт)

1. **`generate-ai-code-stream`**-ийг **чөлөөт промптээс модулийн жагсаалт + InstanceConfig руу шилжүүлэх** (PRD U5/FR5) — үл мэдэгдэх модуль руу **400**.
2. **Sandbox дотор үнэндээ** модулийн директороос файл хуулж, navigation/theme үүсгэх (PRD-д тодорхойлсон дараалал).
3. **Agentийн үүрэг** нь кодыг дангаар зохион биш, **`instance.config`** болон үлдэгдэл copy/deps зэрэгт туслах загварт шахах.

Энэ нь Lovable-ийн “юу ч бүтээе” загвараас салж, **хангалтын ижил давтагдах шинж чанартай** платформ болгоно — гэхдээ **бизнес платформын зорилготой** бол энэ нь үнэн зөв гол зам.

### C. Чат/UI тал — Lovable мэдрэмж

- Нэг чат-д **яс ойлголт**: “Ямар файл өөрчлөгдсөн”, “Юу суулгасан”‑ыг дандаа ил болгох (одоогоор log/UI хэсэгт част байж болно — бүтээгээр нэг жигд болгох санал).
- **Preview‑ийн алдаа** (`previewConsoleErrors` контекстэд дамжуулах гэх мэт) загварыг **генерацийг дахин чиглүүлэх** дэмжих — Lovable-с ойр давталтын эх үндэс.

### D. Инженерийн найдваржуулах давталт

- **Стрим тасралт**, **буцаах retry** загварыг аудитласан баримтыг илүү нэг мөрөн дээр нэгтгэх (generation route-д retry логик байна).
- **Тестийн хамрах хүрээ**: `assemble-modules`-д unit test үлдээсэн нь маш сайн загвар — **generate/apply гол урсгалд** төстэй минимум smoke test санал болгоно (PRD-д E2E дурдсан).
- **API route ашиглалт**: Олон `app/api/*/route.ts` файл client‑оос дуудагдахгүй байна — санхүүгийн “dead surface”‑ийг багасгаж UX/аюулгүй байдлыг хянах (сонголтонд л).

### E. Sandbox болон платформыг тусгаарлах урт хугацааны PRD-д нийцэх байдал

- `PLATFORM.md` дээр заасан network whitelist, unprivileged user, artifact push гэх мэт нь одоогийн локал/Dev орчинтой зөрүүтэй байж болно — **prod руу очихдоо** сайжруулах саналуудыг нэг дор жагсаах.

## 5. “Lovable‑тэй адилхан”‑ыг тоо хэмжээгээр түвшинлэх (KPI санал)

| KPI | Юу хэмжигдэх вэ |
|-----|----------------|
| Preview refresh хийнэ | Чат давталт → iframe амь үзэгдэл шинэчлэгдэх дундаж секунд |
| Build fail rate | Sandbox дээр анхны генераци / дараагийн засварын дараа Vite алдаагаар унтарсан хувь |
| Өөрчлөлт амжилт | Edit горимонд хэрэглэгчийн нэг даалгавар → зорилгот файлд хүрсэн хувь (байршил‑санал засагдсан эсэх) |
| Тасралт | Stream дутуу файл / хаалтгүй файлын тооноос буурах |

## 6. Урт/дундаы/товч хугацааны зураглалын санал

| Хугацаа | Санал болгож буй гол ажлууд |
|--------|-------------------------------|
| **Богино** | Edit intent + search урсгалын audit; preview‑алдаа контекстийг промптдоо сайн ашиглах; хэрэглэгчийн “юу боллоо” ил харагдах UI |
| **Дунджин** | PRD-д нийцсэн assembler-ийн файлууд хуулах үе шатууд; модулийн scaffold; нэг дорх E2E: сонгоно → сборка → preview |
| **Урт** | `PLATFORM.md` container isolation, деплой пайп (`customer-web artifact`), банк/дундын системийн middleware stub‑аас бодит болгох |

## 7. Эх сурвалжууд (репоны дотор)

- `PLATFORM.md` — платформыг юу болгох билээ гэдэг гол баримжаа  
- `.claude/prds/web-builder-phase-1.md` — generate‑ийг assembler болгоно гэж тодотгосон phase  
- `app/api/generate-ai-code-stream/route.ts` — stream + edit agentic давхарга  
- `app/api/analyze-edit-intent/route.ts` — structured search plan

## 8. Товч дүгнэлт

- **Чөлөөт текст‑генераци**‑г Lovable-с ойруулах авцлаар шууд суулгах бол тасралт, чанартайгаар зөрөх эрсдэл өндөр.  
- **Таны платформын оффишиал зорилго** бол **модуль + assembler + shадcn constraint** юм бол — Lovable-с “product shape”‑ээрээ ялгарч болох ч хэрэглэгчийн **хурдавчилсан итрэлт, preview, засварлагдах** зан төлөв нь төстэй байх ёстой.  
- Үүн дээр суурланаар **хаалтлагдмал гаралт** (assembler) ба **нарийвчилсан edit давталт** (intent/search)‑ыг комбиноорол нь мину платформыг зах зээлд **“Lovable маягийн оператор туршлага” + “бизнес‑тай платформ”** гэсэн давхар давуу тал өгөх боломжтой.
